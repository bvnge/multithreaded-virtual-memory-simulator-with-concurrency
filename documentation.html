1. Project Overview

1.1 Introduction

This project is a comprehensive educational simulator suite featuring two powerful applications for demonstrating virtual memory concepts, multithreading, concurrency, synchronization, and page replacement algorithms. It provides interactive Java Swing-based graphical user interfaces where students can learn both single-threaded and multithreaded demand paging concepts through visual simulation.

1.2 Project Objectives

The primary objectives of this project are:

  • To provide an interactive learning environment for operating system concepts
  • To demonstrate page replacement algorithms visually
  • To simulate multithreaded execution with concurrency control
  • To integrate artificial intelligence for intelligent analysis and recommendations
  • To support research and experimental analysis of algorithm performance

1.3 Target Audience

This simulator suite is designed for:

  • Computer Science students learning Operating Systems
  • Educators teaching OS and system programming courses
  • Researchers analyzing algorithm performance characteristics
  • Self-learners exploring operating system concepts

4. Features and Functionality

4.1 Page Replacement Features

4.1.1 Algorithm Support

The simulator implements four page replacement algorithms:

FIFO (First-In-First-Out): Pages are evicted in order of arrival using a simple queue-based implementation. This algorithm has low overhead but suffers from Belady's Anomaly where increasing the number of frames can paradoxically increase page faults.

LRU (Least Recently Used): Evicts the page that has not been accessed for the longest time. This algorithm is excellent for temporal locality and is the industry standard. It has moderate implementation complexity and does not suffer from Belady's Anomaly.

MRU (Most Recently Used): Evicts the most recently accessed page. While counter-intuitive, this algorithm is useful for certain cyclic patterns and serves as an educational comparison point.

OPT (Optimal): Evicts the page that will not be used for the longest time in the future. This is the theoretical best performance algorithm but requires future knowledge and is not implementable in practice. It serves as a performance baseline for comparison.

4.1.2 Interactive Features

  • Step-by-step execution with visual frame updates
  • Automatic playback mode with adjustable speed
  • Color-coded visual feedback for hits and faults
  • Real-time statistics display
  • Algorithm comparison charts
  • Eviction notifications

4.2 CPU Scheduling Features

4.2.1 Scheduling Algorithms

FCFS (First-Come-First-Served): Non-preemptive algorithm with simple implementation. Processes execute in arrival order. This algorithm has low overhead but suffers from the convoy effect where short processes wait for long processes.

Priority-Based Scheduling: Can be preemptive or non-preemptive with higher priority processes executing first. This flexible algorithm can cause starvation of low-priority processes and demonstrate priority inversion issues.

Round-Robin: Preemptive time-slicing algorithm providing fair CPU allocation. Performance is quantum-dependent with good response time characteristics. Best suited for time-sharing systems.

4.2.2 Metrics Calculated

  • Average waiting time
  • Average turnaround time
  • Throughput (processes per time unit)
  • Individual process completion times
  • Gantt chart timeline generation

4.3 Multithreading Features

4.3.1 Thread Management

  • Support for 3-8 concurrent threads
  • Individual reference strings per thread
  • Priority assignment (1-10 scale)
  • Thread state management (RUNNING, READY, WAITING, BLOCKED, COMPLETED)
  • Context switching with overhead tracking

4.3.2 Synchronization Mechanisms

Mutex (Mutual Exclusion): Provides exclusive access to resources using a binary semaphore model. Prevents race conditions but can cause deadlock. Features simple acquire/release operations.

Semaphore: Counting semaphore with configurable permits (1-5). Supports resource pooling and flexible concurrency control. Reduces lock contention compared to mutex.

4.3.3 Deadlock Detection

The system implements Resource Allocation Graph (RAG) cycle detection using Depth-First Search (DFS). Features include:

  • Automatic detection during simulation
  • Visual notification when deadlock occurs
  • Identification of all deadlocked threads
  • Display of circular wait chains
  • Timestamp of detection

4.4 AI Integration Features

4.4.1 Dataset Generation

Reference String Generation: Creates difficulty-based reference strings (Easy, Medium, Hard) with realistic page access patterns considering locality principles. Generates benchmark-quality datasets suitable for algorithm testing.

CPU Process Generation: Creates process datasets with configurable process counts, varied arrival times, realistic burst times, and priority distribution. Output is in CSV format ready for immediate use.

4.4.2 Explanation and Analysis

Page Replacement Analysis: Provides algorithm comparison explanations, performance reasoning, locality impact discussion, and best algorithm recommendations based on workload characteristics.

CPU Scheduling Analysis: Delivers workload characterization, algorithm suitability explanations, quantum impact analysis, and optimization suggestions tailored to the specific process mix.

4.4.3 Output Modes

Concise Mode: Provides bullet-point summaries with quick actionable insights, direct recommendations, and minimal technical jargon. Ideal for quick decision-making.

Research Mode: Generates IEEE-friendly structured reports with sections for Objective, Findings, Evidence, Threats to Validity, and Recommendations. Uses academic language and provides citation-ready, comprehensive analysis.

4.4.4 Anomaly Detection

The AI system detects operating system simulation anomalies including:

  • Thrashing Detection: Identifies excessive page fault rates
  • Starvation Detection: Finds threads that never execute
  • Deadlock Risks: Identifies circular wait patterns
  • Unfair Scheduling: Detects imbalanced CPU allocation
  • Performance Degradation: Identifies excessive context switch overhead

Each detection includes confidence levels, mitigation strategies, root cause analysis, and prevention recommendations.

5. Algorithm Implementations

5.1 Page Replacement Algorithms

5.1.1 FIFO Implementation

The FIFO algorithm uses a queue data structure to track page arrival order. When a page fault occurs and all frames are full, the algorithm removes the page at the front of the queue (oldest page) and adds the new page to the back. Time complexity is O(1) for eviction decisions.

5.1.2 LRU Implementation

The LRU algorithm maintains a list of pages ordered by recency of access. On each page access, the page is moved to the end of the list. When eviction is needed, the page at the front of the list (least recently used) is removed. Time complexity is O(n) for list reordering.

5.1.3 MRU Implementation

Similar to LRU but removes the page at the end of the recency list (most recently used). This algorithm is useful for sequential scan patterns where the most recently accessed page is unlikely to be needed again soon.

5.1.4 OPT Implementation

The optimal algorithm looks ahead in the reference string to find which resident page will be used farthest in the future. This page is selected for eviction. While not practical for real systems, it provides the theoretical minimum number of page faults for comparison purposes.

5.2 CPU Scheduling Algorithms

5.2.1 FCFS Implementation

Processes are sorted by arrival time and executed in order. Each process runs to completion before the next begins. Implementation uses a simple queue with O(n log n) sorting complexity for initial ordering.

5.2.2 Priority Scheduling Implementation

Maintains a ready queue sorted by priority. At each scheduling decision, the highest priority ready process is selected. Uses comparison-based selection with O(n) complexity per scheduling decision.

5.2.3 Round-Robin Implementation

Processes are organized in a circular queue. Each process executes for one time quantum before moving to the back of the queue. If a process completes within its quantum, it is removed from the queue. Implementation uses a queue with quantum counter.

5.3 Deadlock Detection Algorithm

5.3.1 Resource Allocation Graph

The system builds a directed graph where:

  • Nodes represent threads and locks
  • Edges from threads to locks represent waiting relationships
  • Edges from locks to threads represent holding relationships

5.3.2 Cycle Detection

Uses Depth-First Search to detect cycles in the resource allocation graph. A cycle indicates a circular wait condition, which is necessary for deadlock. The algorithm maintains a visited set and a recursion stack to identify cycles in O(V + E) time complexity where V is vertices and E is edges.

8.3 Usage Scenarios

8.3.1 Understanding Belady's Anomaly

Students can observe the FIFO paradox where increasing memory frames increases page faults. Using reference string "1 2 3 4 1 2 5 1 2 3 4 5" with 3 frames yields 9 faults, but with 4 frames yields 10 faults, demonstrating that FIFO does not respect the stack property.

8.3.2 Observing Deadlock

The deadlock scenario demonstrates circular wait conditions. Students configure threads with conflicting lock requirements, observe threads blocking each other, and see automatic deadlock detection with resource allocation graph visualization.

8.3.3 Studying Priority Inversion

Students configure three threads with priorities 10, 5, and 1. With mutex synchronization enabled, they observe the high-priority thread waiting for a lock held by the low-priority thread while the medium-priority thread executes, demonstrating the need for priority inheritance protocols.

8.3.4 Analyzing Thrashing

With six threads and only three frames, students observe excessive page faults, poor hit ratios across all threads, and frequent evictions. AI anomaly detection confirms thrashing conditions and suggests mitigation strategies.

9. Technical Specifications

9.1 Design Patterns

9.1.1 Model-View-Controller

The architecture separates concerns:

  • Models: ProcessThread, LockResource, FrameEntry represent data
  • Views: Swing GUI components display information
  • Controllers: Simulators and schedulers manage logic

9.1.2 Observer Pattern

GUI components observe model changes and update automatically. Event-driven architecture ensures responsive user interface with thread-safe updates.

9.1.3 Strategy Pattern

Algorithms are interchangeable at runtime. Users can select different page replacement or scheduling algorithms without code modification, supporting extensible design.

9.2 Data Structures

9.2.1 Page Replacement

  • ArrayList for frame storage providing O(1) access
  • LinkedList for FIFO queue with O(1) enqueue/dequeue
  • List-based LRU tracking with O(n) reordering

9.2.2 Thread Scheduling

  • Queue for FCFS with FIFO ordering
  • Priority queue for priority scheduling
  • Round-robin queue with quantum counter

9.2.3 Deadlock Detection

  • Graph representation using adjacency lists
  • DFS for cycle detection with recursion stack
  • Resource allocation matrix tracking ownership

9.3 Thread Safety

9.3.1 Synchronization Mechanisms

The system uses Java's synchronized keyword for critical sections. Thread-safe collections prevent concurrent modification exceptions. GUI updates use SwingUtilities.invokeLater() to ensure execution on the Event Dispatch Thread.

9.3.2 Concurrency Control

Lock waiting queues manage blocked threads. State transition management prevents invalid state changes. Race condition prevention through proper synchronization.

9.4 Performance Considerations

9.4.1 Algorithm Efficiency

  • O(n) page replacement decisions
  • O(1) FIFO eviction using queue
  • O(n) optimal lookahead in reference string
  • O(n log n) scheduling with sorting

9.4.2 Memory Management

  • Efficient data structure reuse to minimize allocations
  • Minimal object creation during simulation
  • Event aggregation to reduce memory footprint
